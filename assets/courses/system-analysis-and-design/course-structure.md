   - Introduction to SDLC
   - Four phases: Planning, Analysis, Design, Implementation
   - Seven steps of SDLC
   - SDLC as a process of gradual refinement
   - Deliverables per phase

   - Process-Centred Methodology
   - Data-Centred Methodology
   - Object-Oriented Methodology

   - Steps for designing an interface
   - Control functions: Text Box, Number Box, Selection Box, Buttons
   - Data Validation techniques
   - Data Verification techniques
   - Output Design: On-screen reports, Printed reports

8. **Tools & Techniques for System Development**
   - Data Flow Diagrams (DFD)
   - Entity Relationship Diagrams (ERD)
   - Flowcharts

   - Enhancement specifications

## Definitions

### System
An organized collection of components that work together to achieve a specific or set of goals. Also: a set of interrelated elements that collectively work together to achieve some common goals. Derived from the Greek word *Systema* — an organized relationship amongst functional units and components.
*Source: SYSTEM_ANALYSIS.md*

### Information System
A system designed to manage and process data to produce useful information within an organization. A set of interrelated components that stores, processes and distributes information, and supports decision making.
*Source: SYSTEM_ANALYSIS.md*

### SDLC (Systems Development Life Cycle)
The process of understanding how an Information System can support business needs by designing a system, building it, and delivering it to users.
*Source: SDLC_Lecture_Notes.md*

### Requirement
A statement of what a system must do or what characteristics it must have.
*Source: SYSTEM_ANALYSIS.md*

### Functional Requirement
A requirement that relates directly to a process a system must perform or information it needs to obtain. Flows into functional, structural, and behavioural models.
*Source: SYSTEM_ANALYSIS.md*

### Non-Functional Requirement
A requirement referring to the behavioural properties the system must have — deals with *how* the system will be developed.
*Source: SYSTEM_ANALYSIS.md*

### System Design
The process of planning a new system or replacing/complementing an existing or legacy system. Defines system outputs, inputs, interface, dialogue and data requirements.
*Source: System_Design_Interface_Design.md*

### User Interface
The part of the system users interact with. Includes screen displays, forms for capturing data, and reports the system produces.
*Source: System_Design_Interface_Design.md*

### System Analyst
A professional who specializes in analyzing, designing, and implementing information systems. Acts as a bridge between business problems and technology solutions.
*Source: SDLC_Lecture_Notes.md*

### Waterfall Model
A linear and sequential design process where progress flows steadily downwards; each phase must be completed before the next begins.
*Source: SDLC_Lecture_Notes.md*

### V-Model (Validation & Verification Model)
An extension of the Waterfall model with corresponding testing phases for each development stage; emphasizes early defect detection.
*Source: SDLC_Lecture_Notes.md*

### RAD (Rapid Application Development)
A methodology favouring iterative development and rapid prototyping over large amounts of up-front planning.
*Source: SDLC_Lecture_Notes.md*

### Spiral Model
A model combining iterative development with risk assessment; each iteration involves planning, risk analysis, engineering, and evaluation.
*Source: SDLC_Lecture_Notes.md*

### Methodology
A formalized approach to implementing the SDLC — includes a list of steps and deliverables.
*Source: SDLC_Lecture_Notes.md*

### Legacy System
An outdated computing system including hardware, software, file formats, and programming language.
*Source: System_Design_Interface_Design.md*

## Examples

### System — Examples by Type
- Open System: businesses, ecosystems, computer systems connected to the internet
- Closed System: fully self-contained systems with no external interaction
- Manual System: controlled by humans
- Automated System: controlled by machines
*Source: SYSTEM_ANALYSIS.md*

### System Components — Concrete Examples
- Input: raw data, user commands, energy, materials
- Process: data processing, computation, manufacturing
- Output: processed information, finished product, reports, decisions
- Control: quality control process, management oversight, automated control systems
- Environment: external data sources, regulatory constraints, economic conditions
- Boundary: scope of a project, organizational boundaries, the fence of your house
*Source: SYSTEM_ANALYSIS.md*

### Requirements Gathering — Techniques in Practice
- Interviews: structured/unstructured conversations with stakeholders
- Questionnaires: paper-based or electronic (e.g., Google Forms)
- JAD/RAP: collaborative workshops
- Brainstorming: group idea generation
*Source: SYSTEM_ANALYSIS.md*

### Data Validation vs Verification
- Validation checks format only: `12/11/1928` passes a date format check
- Verification catches wrong values: double-entry would flag `12/11/1928` vs intended `12/11/1982`
*Source: System_Design_Interface_Design.md*

### SDLC Models — Best Fit Use Cases
- Waterfall: projects with well-defined, stable requirements
- V-Model: projects where quality and reliability are critical
- RAD: fast delivery with changing requirements
- Spiral: large, high-risk projects requiring risk management
- Agile: dynamic projects where requirements change frequently
*Source: SDLC_Lecture_Notes.md*
